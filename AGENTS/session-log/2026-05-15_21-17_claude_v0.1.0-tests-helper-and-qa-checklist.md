### `getRuleAttribution` shape

```typescript
RuleEngine.getRuleAttribution(tag, tagMeta, rules) returns {
  tag: string,
  effective: AttributedMatch | null,   // the winning rule (last-match-wins)
  allMatches: AttributedMatch[],        // every matching enabled rule, priority desc
}

AttributedMatch = {
  ruleId, ruleName, action, scopes, priority, builtin, reason
}
```

Reasons render as:
- regex: `matches /^[A-Fa-f0-9]{3,8}$/`
- frequency: `count 3 <= 1`
- list: `exact match in list`

The tag list view consumes `effective.reason` for the diagnostic column and `allMatches` for an optional "rule chain" expander.

## Decisions

**1. Vitest over Jest.**
Faster, native TypeScript, simpler config, no Babel. The codebase has no Jest history to preserve.

**2. Vitest 3.x, not 2.x.**
Started on 2.1.9 to match the plan-era stability profile, but the Vite CJS deprecation warning was noisy. Upgraded to 3.2.4 and renamed `vitest.config.ts` to `.mts`. Warning gone. Same test API, no migration work.

**3. `obsidian` stub via Vite alias, not `vi.mock()` per file.**
Centralizes the substitution and means no boilerplate in each test file. Resolves at import time, which matches how the production code reads.

**4. Tests in `tests/`, not colocated.**
Keeps `src/` clean for both the production build (esbuild bundles main.ts) and the lint pass (`src` only). Trade-off accepted: no auto-discovery for IDE "jump to test" shortcuts.

**5. Fix the schemaVersion bug at source, not in the test.**
The test caught a real defect: `migrate()` always sets `merged.schemaVersion = SCHEMA_VERSION`, so the `if (this.settings.schemaVersion !== SCHEMA_VERSION)` guard never fires and `persist()` is never called from `load()`. A v0 install that doesn't make subsequent user-action saves keeps a v0 `data.json` on disk forever. Fix: capture `incomingVersion = (raw.schemaVersion ?? 0)` BEFORE `migrate()` and compare against that.

**6. `getRuleAttribution` returns richer types than `evaluateTag`.**
Existing `MatchResult` is `{matched, ruleId, ruleName}`. The new helper needs `action`, `scopes`, `priority`, `builtin`, and `reason` so the UI can render the full diagnostic. Added new `AttributedMatch` and `RuleAttribution` types rather than expanding `MatchResult` (which might be relied on elsewhere). No churn in existing code paths.

**7. "Last-match-wins" verified to mean LOWEST priority among matches.**
The implementation sorts rules priority-desc, iterates, overwrites `lastMatch` on every match. So the last-set value is the LAST rule in priority-desc iteration = the LOWEST priority. Test `tests/ruleEngine.test.ts:42` locks this; the existing comment in `ruleEngine.ts` said "Last match wins" without defining "last," so the test documents the intent.

**8. Did not add observer or main.ts tests.**
Both need JSDOM environment + mocking of `App`, `Plugin`, `WorkspaceLeaf`. Material setup cost for code that is primarily DOM glue around the now-fully-tested engine. Phase C will likely change observer integration points anyway. Defer to v0.2.

**9. Did not push.**
Five new commits sit on `release/v0.1.0` ahead of origin. User instruction was "just go" on the work, not on pushing. Stopping at clean local commits and surfacing the state.

## Notes for Future Iteration

- The test sweep paid for itself on the first run: it caught the schemaVersion persistence bug that would have shipped silently. "Tests before UI rewrite" was the right order specifically because the legacy UI was the only consumer triggering persists often enough to mask the bug.
- "Plan says partially done" vs "code says done" drift is real. Always cross-reference recent commits against open plan items before writing a status report. The plan is authored before code lands; commits are the source of truth for "is this done."
- The `obsidian` module stub is intentionally tiny. Resist the urge to mock the whole API up front. Each test that needs more should expand the stub for its case.
- Renaming `vitest.config.ts` to `.mts` is the cleanest way to silence the Vite CJS deprecation warning without adding `"type": "module"` to package.json (which would break `.eslintrc.cjs` resolution conventions). File-extension wins.
- The `AttributedMatch` shape was designed for the diagnostic UI to be a pure render of the helper's output. The UI should not re-derive priority order, re-evaluate matchers, or look up rule details from the settings object. If a future UI needs more (e.g., links to the rule editor), add fields to `AttributedMatch` rather than introducing UI-side lookups.
