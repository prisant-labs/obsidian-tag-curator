It closed the v0.1 design loop after seven rounds of crit-driven review, landed the full v0.1 UI in source code across four phases, fixed a silent engine bug that would have shipped if undetected, renamed `dryRun` to `previewMode` end-to-end on Domain-Driven Design grounds, reconciled the entire discovery and planning corpus to match what is actually shipping, and produced three full feature-proposal bundles (spec + plan + UI mockup) for v0.2 work. ## Decisions Made

This session produced or revised 14 D-IDs and 8 Q-IDs in `scope-and-decisions.md`. The architecturally significant ones get full analysis; the smaller ones are noted briefly.

### D-001 - Rule editor IA, evolved through three layouts (architectural)

The rule editor went through three layouts in this session, each replacing the prior in response to crit feedback. The trajectory matters because each transition resolved a specific class of concern.

- **Layout A (initial converged).** Inline expand-row with a 2-column field grid (Name/Priority, Match type/Pattern, Action/Scope) and a pill cloud preview. Reviewer flagged poor IA (no logical top-to-bottom process) and unscannable pills.
- **Layout B (D-001 sectioned vertical).** Recommended in response; kept inline but reorganized as Identity / Match / Then / Preview with a scannable list (not pills). Reviewer accepted the IA fix but pushed back on expand/collapse layout shift.
- **Layout C (D-001 evolved to master-detail).** Two-pane: rule list on the left, sectioned editor on the right. Selection drives the detail. Resolved layout shift but reviewer asked for an alternative: "card view that clicks into edit mode" with a "right-docked, scrollable preview."
- **Layout D (D-010, locked).** Card view + right-docked preview. Each rule is a full-width card. Clicking enters edit mode. The right-docked preview is persistent across card and edit views. This is what shipped.

The lesson, recorded in memory as `honor-feedback-specifically`: when the user's feedback names a specific UI critique, the right response is to redesign the surface, not to bucket the critique into "kept / deferred / acknowledged."

### D-002 - Wizard dropped (architectural)

The original spec described a multi-step first-run wizard. Variant explorations in earlier rounds proposed wizard-then-master-detail for "new rule" creation. Round-3 review pin 17 argued the rule editor IA is already learnable enough that a wizard adds unnecessary complexity. D-010's card view + Type dropdown handles creation without a separate flow. D-002 closed; the wizard is gone.

### D-003 - dryRun -> previewMode (DDD ubiquitous-language, architectural)

The original feature was named `dryRun` in code and "Preview mode" in user-facing UI. I initially proposed a split: rename UI label only, keep the internal key to avoid a migration. User challenged this on Domain-Driven Design grounds: the Ubiquitous Language principle says one name across domain, UI, and code. The split was the translation tax DDD is designed to eliminate.

Reversed the call. Renamed end-to-end: `previewMode` in types, settings, observer, main, settingsTab, command id, status bar, Notice copy. Schema bumped v1 -> v2 with automatic migration. The pre-1.0 timing made migration cheap; deferring would have been the more expensive choice.

### D-004 - Merge/alias is display-only (decided 2026-05-28 morning)

Confirmed early in the session. Carried into the aliases proposal as a foundational constraint.

### D-005 - Built-in presets are toggleable, not editable

Confirmed early; aligned with what the code already does.

### D-006 - Tag list is row-based, not drag-drop

Confirmed early; aligned with the existing tag list table approach.

### D-007 - Persistent state banner (architectural)

Round-2 pin 2 surfaced the ambiguity that panic disable is both an action (DOM cleanup) and a state (plugin off). Same for preview mode: easy to forget you are in it. Decided on a persistent top-docked banner above every Tag Curator surface whenever the plugin is in a non-default state. Two variants: `Preview mode is on` (amber) and `Tag Curator is off` (muted). Each carries an inline action to restore default. Implemented in `src/ui/stateBanner.ts`.

### D-008 - Welcome modal redesign (significant)

Round-2 pins 5-8 flagged four problems with the initial welcome modal: state-unaware header ("Got it, enable" when plugin is already on), centered chunky safety-promise cards, wrapped-paragraph integration rows, buried preset section. Reworked end-to-end: state-aware eyebrow + h3 + sub; safety promises as a single highlighted strip of left-aligned check rows; toggleable preset cards; per-plugin integration cards with state pills + bullet lists; primary "Start curating" + secondary "Start in preview mode" CTAs with inline explainer.

### D-009 - Priority architected, hidden from UI for v0.1 (architectural)

Round-3 pin 11 raised the question: should priority be architected in the engine but not visually functional? Four options considered: (A) keep visible integer priority, (B) hide it in UI with engine-side architecture, (C) drag-to-reorder UI, (D) simpler 1-5 or 1-10 scale. User confirmed B: architect it with documentation, remove from UI. Custom rules default to `priority: 50`. v0.2 surfaces drag-to-reorder (B012). Implementation: master-pane / card-view drops the priority chip; the Identity section drops the Priority row.

### D-010 - Card view + right-docked preview (architectural, supersedes D-001)

Round-3 pin 12 + pin 17 together. Reviewer asked for "a right docked, scrollable panel" preview and "a card view that clicks into edit mode" instead of left-docked master-detail. Implemented in Phase 4: full-width cards with click-to-edit, persistent right-docked preview panel (vault-wide in card view, filtered to the selected rule in edit mode), stats row (Unique tags / Total instances / Notes touched) in the preview head. Supersedes D-001; D-002 closes as a consequence.

### D-011 - Tag list as both sidebar leaf and Settings tab (significant)

Round-3 pin 21 asked whether Tag list should be a Settings tab. I initially recommended keeping it as a sidebar leaf only with a discovery button in General. Reviewer pushed back: "How does it double up two surfaces?" Re-examined and found the doubling concern did not hold. Tag list is one component; state lives on the plugin, not the view. Multiple hosts render the same data. Reversed: ships as both. The sidebar leaf and the Settings tab render the same component; state stays in sync.

### Q-005 - rule precedence bug, fixed (significant correction)

While answering pin 9 (multi-rule attribution), I read the engine carefully and found the precedence implementation sorted priority-descending then kept the LAST match in the loop, which inverted "priority" semantics to "lowest-priority wins." Earlier I had told the user "highest priority wins"; that was wrong. Honest correction in the conversation, then fixed: return the first match in priority-desc order, set `effective = allMatches[0]`. Tests updated to lock the corrected semantics. Committed with D-003 because the two changes naturally co-existed in the same touchpoint set.

### Q-007 - Tag list as Settings tab (closed as D-011)

Reversed mid-session as described above.

### Q-008 - File-extension as match type vs file-filter on rule (closed)

Round-3 pin 15 asked whether file-extension should be a match type or a parallel concept. I initially leaned toward "new match type"; reviewer's follow-up reframed it as "could be a match rule, not a match type." Re-examined: a match type is `how-to-match-the-tag`; file-extension is `which-files-to-consider` - different axes. Reversed to: `Rule.fileFilter?: { extensions?: string[] }` parallel to match type. Carried into the v0.2 proposal as B011.

### D-012 (not formally numbered): promote B006 aliases from v0.3 to v0.2

When asked "what are the most valuable new features," I re-ranked the backlog by user-pain-solved-per-week-of-work rather than by spec roadmap. Aliases ranked #1; recommended promotion from v0.3 to v0.2. User did not formally commit but accepted the recommendation through the proposal-bundle creation. Captured in the proposals/README and the GitHub milestone assignment (#11 now in v0.2.0).
