## Decisions

**1. Top tabs replace left sidebar for the plugin's internal settings navigation.**
Why: Obsidian's outer Settings UI already uses left-nav for the plugin list. Adding more left-nav inside a plugin's settings double-stacks the navigation pattern and reads as competing chrome rather than within-plugin sub-navigation. Top tabs avoid that conflict and match the pattern users know from VS Code, JetBrains, and modern web settings. The user explicitly directed this; the prior Option 2 (sidebar nav, matching the mockup PNG) was replaced rather than supplemented.

**2. Each new option section gets a single focused mockup, not multi-variant.**
Why: The existing 7 options each had one focused mockup (with Option 7 status-bar as the exception because 4 sub-variants are small enough to be useful). Adding 14-24 new sections with multi-variant mockups would balloon the doc to 8,000+ lines and harm scannability. Each new section sticks to one focused mockup with clear pros/cons.

**3. Engine-extension flag system introduced in round 3 to gate scope.**
Why: Round 2 (ecosystem research) made it clear that some "potential experience" options are pure UI on the existing engine (Options 23, 24, 26, 28, 29, 30) while others require data model changes (22, 25, 27, 31). An orange pill in the section title makes this visually obvious so users can scope-gate ruthlessly: ship UI-only options first, batch engine extensions into v0.2+ milestones. Without this flag, the doc reads as a flat list of equally-priced options, which would have led to bad scope decisions.

**4. Ecosystem research lives in `discovery/`, not `release-plans/`.**
Why: The repo already had three discovery docs (spec, implementation plan, project overview) at `docs/internal/discovery/`. Those are the inputs that inform planning. The new ecosystem research has the same shape - it's research that feeds the next plan iteration. Putting it next to the spec creates a coherent body of "things the plan was based on" that future agents can read in one pass.

**5. Style Settings deep-dive treats it as a critical (not optional) integration.**
Why: Style Settings is the de facto extension point for theme customization in Obsidian. The research found that it auto-discovers `/* @settings */` blocks in any plugin's CSS, requires zero JavaScript from the plugin side, and costs about 50 lines of CSS comments. The leverage ratio is high enough that recommending against it would only be defensible if Style Settings were unmaintained or controversial - neither is true. The research recommends Tag Curator should NOT depend on Style Settings being installed (the defaults work standalone), but SHOULD expose the variables.

**6. "Non-engineer UX gap is at rule creation" is the headline UX risk.**
Why: Every popular plugin that exposed power-user logic without a GUI builder (Tasks v1, early Dataview, raw Templater) saw stalled adoption. Plugins that added a wizard or examples library (Tasks v2, Templater snippets) saw adoption climb. The research positions the wizard (Option 13) + examples library + welcome modal (Option 17) as the three-piece UX scaffold that lets Tag Curator avoid the v1-Tasks fate. This is now the explicit framing for Phase C scoping.

**7. Three plugin integrations are critical, not just the two the user named.**
Why: The user said "in addition to Tag Wrangler and Notebook Navigator." Research adds Style Settings as the third critical (because it's the customization extension point) and identifies 6 high-value plugins requiring compatibility audits (Iconize, Various Complements, QuickAdd, Dataview, Tasks, Colored Tags Wrangler). The 12 compatible-by-default plugins are flagged in the research doc as "verify, document, do not integrate" - the display-only contract preserves them.

**8. Compound criteria (Option 22) requires engine extension; deferred to v0.2.**
Why: The v0.1 `MatchCriteria` is a flat object with one criterion type. Compound logic (AND/OR/NOT) requires changing it to a tree of `MatchNode` objects, plus an evaluator that walks the tree with short-circuit logic. The migration is trivial (existing rules become single-child trees) but the engine work is real. Scope: v0.1 ships the flat model and the wizard for novices; v0.2 adds the compound canvas for power users without breaking v0.1 rules.

**9. Co-occurrence rules (Option 27) deferred because of data shape, not UX.**
Why: Per-tag co-occurrence data is quadratic in tag count. A vault with 1,000 tags would have a million potential pairs. Mitigation strategies (lazy compute on rule-eval, cache, file-change invalidation) work but are non-trivial to get right. v0.2 is the earliest reasonable ship; v0.3 is safer.

**10. Tag review workflow (Option 28) reuses the existing `TagMeta.reviewed` field.**
Why: The plan already added `reviewed?: boolean` to `TagMeta` for future use. The workflow proposed extends this to three states (pending, decided, ignored) without changing the type's shape beyond a serialization detail (boolean vs enum). No engine cost. Pure UI work. Could plausibly ship in v0.1 if the bulk-action toolbar (Option 10) lands too.

**11. Merge & alias (Option 31) treats Tag Wrangler as the rename path.**
Why: Tag Curator's file-safe contract means it cannot rename tags in notes. The merge & alias workflow shown is display-only collapse: multiple alias tags fold into one canonical display row. The mockup explicitly tells users to install Tag Wrangler if they want actual rename. This is the cleanest division of responsibility: Tag Curator owns display semantics, Tag Wrangler owns file mutation.

## Notes for Future Iteration

- The session structure (round 1 expansion -> research doc -> round 2 expansion) ended up being the right order in hindsight. Round 1 explored what was possible; the research doc grounded what was sensible; round 2 incorporated the research findings (especially the "engine extension" gating concept and the v0.2-v0.4 vision framing). If the order had been reversed - all expansion first, research last - the engine-extension flag would not have made it into the doc, and the picks would have read as flat-priced. Doing the research between expansions is the pattern worth repeating for future exploration sessions.

- 31 sections is the upper bound of "skimmable mockup picker." Past 35-40 sections, split into per-area HTML files with a top-level index. The current sidebar nav works because section labels group them into 9 areas, but past 40 the eye loses the area boundaries.

- The "engine extension" pill (orange, top-right of section title) is a generalizable pattern. Any time a design doc mixes pure-UI work with data-model changes, marking the latter visually saves scope decisions later. Worth porting to other product surfaces.

- The ecosystem research doc has a Part 8 "Risks and unknowns" section that flags things to verify before implementation. This is a pattern worth porting to future research docs - it separates "I'm confident about this" from "I'm assuming this but verify before betting on it." Without the section, readers conflate the two.

- The "two-tier creation funnel" pattern (wizard for novices, expert path for power users) is the single highest-leverage UX insight from the ecosystem research. It applies to Tag Curator (wizard + inline + builder) but also to any future plugin that exposes power-user logic. Worth keeping in the design-pattern memory.

- Style Settings as an integration pattern is more elegant than building a custom theme. Tag Curator does not depend on Style Settings; it just declares variables. Themes get a free customization point. Users get a GUI if Style Settings is installed; otherwise the defaults work. This is the cleanest plugin-to-plugin integration model in the Obsidian ecosystem and worth porting to other plugin work.
