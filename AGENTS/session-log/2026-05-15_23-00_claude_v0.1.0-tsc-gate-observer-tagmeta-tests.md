### Source bugs fixed

**Bug 1 (caught by self-review)**: `src/storage/settings.ts:28` previously used `incomingVersion !== SCHEMA_VERSION`. For a future v2 file read by a v1 plugin, this would trigger persist and overwrite v2 data with v1-shape defaults. Now `incomingVersion < SCHEMA_VERSION` so older plugins reading newer files leave the on-disk data intact.

**Bug 2 (caught by tagMetaManager.test.ts)**: `src/storage/tagMeta.ts:75-78` previously did `const source = inlineSet.has(tag) ? 'inline' : 'frontmatter'; this.touchTag(tag, source, now);`. For a tag in both inline body AND frontmatter, only `'inline'` was ever recorded. Fix:

```typescript
const seen: TagSource[] = [];
if (inlineSet.has(tag)) seen.push('inline');
if (frontmatterSet.has(tag)) seen.push('frontmatter');
if (seen.length === 0) seen.push('frontmatter'); // defensive fallback
for (const source of seen) this.touchTag(tag, source, now);
```

The `touchTag` merge logic was already correct (`if (!existing.sources.includes(source)) existing.sources.push(source)`); the bug was that it never got called twice for the same tag in a single indexFile invocation.

## Decisions

**1. `tsc --noEmit` runs only against `src/`.**
The tsconfig include is `["src/**/*.ts"]`. Tests live in `tests/` outside that scope. This is intentional: the gate exists to catch the Path A class of bug where esbuild strips type errors silently from production code. Tests have a parallel safety net (vitest fails on runtime errors). Adding tests to the include would force the stub to be production-grade and would surface vitest-runtime type noise.

**2. happy-dom instead of jsdom.**
jsdom 27.x had a transitive CJS interop bug with `@csstools/css-calc` on Node 20 (`require() of ES Module ... not supported`). happy-dom is lighter, has no ESM/CJS issues, and is vitest's recommended option for DOM tests.

**3. Per-file `@vitest-environment happy-dom` directive.**
Avoids forcing all tests into a heavier DOM environment. The engine and storage tests stay on node (faster, isolated). Only `tagPaneObserver.test.ts` opts in.

**4. Mocked `requestAnimationFrame` via `queueMicrotask`.**
The observer uses rAF to coalesce DOM writes. happy-dom provides a rAF but the timing is inconsistent across versions. Replacing with a microtask gives deterministic flushing per-test (`await flushRaf()`).

**5. Fixed the source bug rather than adjusting the test.**
The `sources` field is plural and `touchTag` has merge logic, both of which are strong signals that "all current sources" is the intended semantic. Adjusting the test to expect `['inline']` would have documented the bug as correct behavior. Fixed the source.

**6. Defensive fallback in the indexFile fix.**
`if (seen.length === 0) seen.push('frontmatter')` handles the theoretical case where Obsidian's `getAllTags` returns a tag from a source neither in inline nor frontmatter (future Obsidian versions may add new tag origins). Real-world impact today: zero. Cost: one line. Worth it.

**7. Skipped main.ts lifecycle tests.**
Would need App / Notice / addCommand / WorkspaceLeaf.setViewState mocking, plus scenario coverage for the 6 commands and `onExternalSettingsChange`. Would be the largest test file in the project. Phase C may rewrite `main.ts`'s command set. Deferred to v0.2.

## Notes for Future Iteration

- "Bug 2" is the second case this 24-hour window where a test added for coverage caught a real defect on first run. Pattern: tests force you to articulate intent, the intent contradicts the code, the contradiction surfaces as a failing assertion. The engine + storage layers were both written quickly to unblock the plugin; their tests are where the intent gets locked in.
- happy-dom is the right default for new Obsidian plugin tests. jsdom's CSS pipeline brings in ESM-only sub-dependencies that don't interop cleanly with vitest's node loader. Reach for jsdom only if happy-dom is missing a specific API.
- Per-file `@vitest-environment` directives are the cleanest way to mix node + DOM tests without paying the DOM startup cost on tests that don't need it. The 8-file suite runs in 1.7s including DOM setup; without the directive it would be substantially slower.
- The stub-expansion pattern is working well: each new test file adds exactly what it needs to `tests/_stubs/obsidian.ts`. After this round the stub is 124 lines and covers most of what production code imports. Future tests for `main.ts` will need to add Workspace methods (revealLeaf, getRightLeaf, getLeavesOfType) and Modal but the pattern stays the same.
