# Session: v1.0 IA Rework (Phases 1-3) + Three Release Candidates

## Goal

The session opened on a BRAT problem ("it doesn't seem to have any of the changes from the last
session") and grew into the full v1.0 information-architecture rework. Three arcs, in order:
(1) diagnose why BRAT showed stale code and ship the missing work; (2) take the user's live-vault
UX feedback and co-design a new IA (curation as a first-class Settings tab + an opt-in pane with
View/Manage modes + a real triage inbox), crit-approve it, and write a phased plan; (3) execute
that plan task-by-task with subagent-driven development plus a Codex adversarial review at each
phase gate, cutting a release candidate per phase.

## Summary

BRAT installs from GitHub *releases*, not branches, and the published `1.0.0-rc.1` had been built
from `cf14e18` - BEFORE the entire Phase 1-8 UI-review work from the prior session landed. So the
fix to the user's first complaint was not code at all: re-cut a higher version. That produced
`1.0.0-rc.2`, which the user then BRAT-tested and reacted to.

That reaction became a real IA redesign. Across four revisions of a single interactive idea board
(`docs/internal/release-plans/plan_v1.0.0/ideas/ui_curate-tags-ia-and-modes.html`, crit-approved
with zero comments) we converged on a "two surfaces, two jobs" model: a **Curate Tags** settings
tab (always full Manage) replaces the dead "Open Tag Curator" button; the dockable pane becomes
**opt-in** via a `paneEnabled` toggle and gains a **View / Manage** mode switch (View navigates by
turning tag names into searches, Manage curates with full per-row + bulk parity); and the
previously-stubbed **Unreviewed** filter becomes a genuine triage inbox via a persisted `reviewed`
flag and Mark-reviewed actions. Settings tabs were also split (Presets / Custom rules) and tidied
(compact Help, stats-on-top, chips-below-search, a new Shown filter).

A 3-phase plan was written with the writing-plans skill (`plan_v1.0.0/plan_v1.0.0.md`) and executed
with subagent-driven development: every one of 17 tasks went implementer -> spec-compliance review
-> code-quality review, with fix loops. The user directed "push through phase 2 and 3 with a Codex
adversarial review at each phase gate," so the BRAT-test pause between phases was replaced by a
Codex pass that found five genuine issues (one O(N^2) perf bug, four correctness/lifecycle bugs),
all fixed. The session ends with `1.0.0-rc.4` published (bundling Phases 1-3), 267 tests green,
everything pushed, and GA the only remaining (user-gated) step.

## Key Decisions

- **BRAT serves releases, not branches; re-cut a higher version to ship new code.** The whole Arc-1
  fix. `main.js`/`styles.css` are gitignored, built only by `release.yml` on tag push. "Code is on
  the branch" and "code is installable" are different states; the bridge is a new tag. Re-uploading
  assets onto an existing tag would not work either (BRAT keys updates off the version string).
- **Two surfaces, two jobs.** Curate Tags settings tab = a deliberate destination, always Manage
  (no view mode - Settings is inherently a manage context). The pane = ambient, opt-in, with the
  View/Manage toggle. View *navigates* (tag name -> Obsidian search for that tag's notes, like the
  native tag pane) and Manage *curates* (full per-row + bulk parity). Giving View a real job stops
  it from being a pointless passive mirror that everyone abandons for Manage.
- **The pane is opt-in via `paneEnabled` (default true).** Default true so existing rc testers do
  not lose the pane on upgrade. The toggle gates the ribbon icon and the two dedicated open-pane
  commands and closes open pane leaves when turned off.
- **Do NOT gate `openCurationWorkspace`/`openBesideTagPane` themselves.** Only the ribbon and the
  two dedicated commands are gated. The Presets "N tags affected" deep-link and the status-bar
  click call `openCurationWorkspace` directly and must keep working. (Deviated from the plan's
  original "no-op when disabled" to avoid regressing the deep-link.)
- **Build the Unreviewed triage inbox for v1.0** (vs cut). `reviewed` persists in the `tags.json`
  sidecar via `setReviewed`/`setReviewedBulk`; survives a rescan for still-present tags; a count->0
  deletion drops it (matches the store-only-live-tags semantics).
- **Codex adversarial review replaces the BRAT-test pause between phases** (user instruction). It
  reaches what per-task spec/quality reviews cannot: cross-cutting, runtime behavior (both surfaces
  open at once; the cost of bulk-marking 1000 tags).
- **Subagent-driven execution with two-stage review per task**; controller provides full task text
  (subagents never read the plan); cheap models for mechanical tasks, sonnet for integration.
- **rc cuts are user-gated** (outward-facing tag push); GA stays a separate explicit step.

## Problems and Solutions

- **Open-button looked dead.** Settings modal covers the opened pane. Interim fix: close Settings
  in the launcher (`9705f66`). Final: the launcher is removed entirely (curation is the Curate Tags
  tab), so the bug class is gone (`6f6dfdb`).
- **Bulk Mark reviewed was O(N^2).** Each `setReviewed` fired `changed`; the view repaints per
  event. Codex Phase-2 finding. Fix: `setReviewedBulk` mutates all present tags, skips unchanged,
  fires ONE save + ONE `changed`; `markReviewed` makes a single host call (`9369113`).
- **Rule filter bled into the status-bar hidden view.** `setRuleFilter` set a filter that
  `setHiddenOnly` never cleared. Codex Phase-3 F-3. Fix: `setHiddenOnly` calls
  `model.setRuleFilter(null)` (`e299e2b`).
- **Curate settings-tab table did not live-update.** The pane subscribes to settings/metadata
  events; the settings-tab table did not. Codex Phase-3 F-1. Fix: subscribe in `renderCurate` with
  a balanced `teardownCurate()` (unsubscribe + `offref` + destroy) called at the top of `display()`,
  in `hide()`, and at the start of `renderCurate` - exactly one live subscription pair when shown,
  zero after (`e299e2b`).
- **RuleEditor leaked subscriptions on tab switch** (pre-existing). Codex Phase-3 F-2. Fix: track
  the instance, destroy before re-create and in display()/hide() (`e299e2b`).
- **Redundant renders on pane mount.** Constructor refreshed, then `setMode` refreshed, then an
  explicit refresh (2-3 renders). Quality-review finding. Fix: pass `initialMode` to the TagTable
  constructor so it builds once in the right mode (`e91c065`).
- **Row-menu Mark reviewed was fire-and-forget.** Quality-review finding: the override items
  `await` + `requestRefresh`; the reviewed toggle did not (would not repaint in the Phase-3
  settings-tab host). Fix: match the override idiom (`cbc5dc3`).
- **View-mode column misalignment risk.** Rows use CSS grid with an 8-track template; `display:none`
  on the select+actions cells would leave orphaned tracks. Fix: a view-mode 6-track grid template
  (the original middle six) applied to both head and rows; hidden cells do not consume tracks.
- **PowerShell here-string in the Bash tool** polluted a commit message with a leading `@`. Fix:
  re-author with separate `-m` flags (the Bash tool is bash, not PowerShell).
- **`&mdash;` HTML entities** slipped into a mockup (renders as an em-dash). Swept and replaced; the
  no-dash hook only catches the literal U+2014/U+2013, not entities, so HTML needs a manual sweep.
