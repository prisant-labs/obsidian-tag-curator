Settings tab · Option 2

Top tabs

Horizontal tab strip across the top of the settings tab, with the active section rendered below. Obsidian's outer Settings UI already uses left-nav for the plugin list, so left-nav inside a plugin's settings double-stacks navigation. Top tabs avoid that and read as a within-plugin sub-navigation rather than a competing chrome layer. Tabs carry small badges with section state (rule count, hidden-tag count) for at-a-glance status.

ProsDoesn't compete with Obsidian's left-nav chrome. Tab badges expose state without an extra row. Scales as v0.2 / v0.3 sections arrive. Familiar pattern from VS Code, JetBrains, modern web settings.
ConsSlightly more custom CSS than Option 1. Tabs can overflow horizontally if section count grows past 7-8 (mitigation: horizontal scroll, established here).

Diagnostics · Option 7

The status bar is the most-seen Tag Curator surface. Spec §8.3 calls it out specifically as the "where did my tag go" prevention. Here are four polished phrasings paired with different click-affordances. Pick one; do not stack them. The v0.1.0 plan ships variant B by default.

ProsAlways visible. One click opens the hidden-tag filter. Conveys plugin state without occupying space elsewhere.
ConsStatus bar text is constrained. Mobile lacks status bar - this surface is desktop-only.

Settings tab · Option 8

Inline settings search

A persistent search field above the settings tab content that highlights matching Setting() rows and hides everything else as the user types. Native Obsidian's outer settings panel uses this pattern; mirroring it inside Tag Curator's settings shortens the path from "I want to change X" to "I'm changing X." Works orthogonally to any layout (Native, Top tabs, Cards) - the search is a layer above the layout.

ProsMatches Obsidian's own settings UX. Scales to v0.3+ when section count grows. Discoverable - users already look for a search field. Negligible CSS cost.
ConsAdds a row of chrome that some users will find unnecessary at v0.1 scale. Empty-search-result state requires its own design.

Onboarding · Option 17

First-run welcome modal

A single, dismissible modal that fires the first time Tag Curator is enabled in a vault. Explains the "display-only, file-safe, fully reversible" contract up-front (the biggest concern users have on install), shows the two default presets that will run, and offers a "preview before applying" path that switches to dry-run mode for first impressions. The plan defers full onboarding to v0.2, but this minimal modal is small enough for v0.1 and answers the most common install-time question.

ProsResolves the trust question immediately. Sets the dry-run habit early. Single dismiss; never appears again. Low implementation cost (one Modal subclass).
ConsModal interrupts the install moment. Some users will skim and miss the file-safe contract anyway. Adds an "already seen welcome" flag to settings.

Welcome to Tag Curator

Tag Curator hides noisy tags from your tag pane without touching your notes. Two things to know before you start:

1

Your notes are never modified.

Tag Curator only changes how tags display in the tag pane. Tags stay in your notes exactly as you wrote them. Uninstall the plugin and everything reverts instantly.

2

Two presets are on by default.

Tag Curator will hide hex color codes (like #FFAA00 from web clippings) and URL anchor fragments (like #section-3). Three other presets exist but are off; turn them on later if useful.

3

Preview before applying.

Start in dry-run mode to see what would be hidden, before anything disappears. We recommend this for the first few minutes.

Onboarding & states · Option 19

Error states

Three failure modes Tag Curator can hit, each with explicit recovery paths. Regex compile failure (most common, in rule editor), tags.json sidecar corruption (rare but possible after disk issues or sync conflicts), and rule conflicts (same tag matched by two rules with the same priority). Each state shows what happened, what's safe, and the one button that fixes it.

ProsFailure modes have explicit owners and recovery actions. Users do not have to guess. The "what's safe" line preserves trust in the plugin's file-safety contract.
ConsThree alert styles to design. Conflict warnings need rule-attribution awareness.
Error states
A. Regex parse failure in rule editor
!
Invalid regex: [unterminated The pattern doesn't compile: unterminated character class at position 1. Fix the pattern or switch to a list-match rule.
B. tags.json corrupted (after a sync conflict, partial disk write, or manual edit)
!
tags.json could not be read The tag metadata sidecar at .obsidian/plugins/tag-curator/tags.json is missing or corrupted. Tag Curator is using an empty in-memory cache until you rebuild. Your notes are safe - this file is auto-generated and contains no original content.
C. iOS-unsafe regex (lookbehind rejected at compile time)
!
Pattern not allowed: lookbehind (?<=draft-)\w+ uses a lookbehind assertion, which crashes plugin load on iOS < 16.4. Rewrite without lookbehind, or use a list-match rule.
D. Two rules at same priority match the same tag
i
Rule conflict on #FFAA00 "Hide hex color codes" and "Custom: hex shorthand" both match this tag at priority 100. The later-defined rule wins under last-match-wins, but the ordering is implicit. Set distinct priorities to make the resolution explicit.

Advanced rule composition · Option 22

Compound criteria builder engine extension

v0.1 rules have one match criterion (regex, frequency, or list). Power users want compound logic: "regex match AND count <= 5 AND NOT in this allow-list." This option adds an AND/OR/NOT group canvas where users compose multiple criteria with drag-and-drop, visual grouping, and per-group operators. Requires extending the rule model from a single MatchCriteria to a tree of criterion nodes.

ProsSolves real cases (hide archived tags that are also orphans). Visual model maps cleanly to the data shape. Composes with the wizard (Option 13) - wizard emits a single-criterion tree, this canvas adds branches.
ConsEngine model change: MatchCriteria becomes MatchNode with op: 'and' | 'or' | 'not' and children. Migration required. Defer to v0.2 once the v0.1 contract is stable.
