Settings tab · Option 1

Native (minimal)

A pure stack of Setting() rows grouped by setHeading(). This is what the v0.1.0 plan ships by default in Task 11. It looks like every other Obsidian plugin and passes the community plugin directory review without modification.

ProsLowest implementation cost. Familiar to all Obsidian users. Zero rejection risk. Theme-agnostic.
ConsVisually flat. No stats glanceability. Harder to find a specific rule in a long list.

Settings tab · Option 3

Cards + stats

Glance-first layout. Stats row up top answers "is the plugin doing anything." Presets are tappable cards instead of toggle rows. Custom rules and Advanced collapse below the fold. Best for marketing screenshots and first-run impressions.

ProsDemos well. New users immediately see impact. Stats reduce the "where did my tag go" fear.
ConsMore CSS surface, more rejection risk in directory review. Less Obsidian-native.

Tag list view · Option 4

Compact table

Dense, sortable, searchable. Each row carries everything needed for triage. Status column attributes hidden tags to the rule that hid them - core to the "Why is this tag hidden?" diagnostic from spec §5.9. This is what the v0.1.0 plan builds in Task 10.

ProsBest information density. Familiar pattern. Fits in a right-sidebar leaf at narrow widths.
ConsPer-row actions need a hover affordance. Bulk operations need a separate toolbar.

Tag list view · Option 5

Master / detail

List on the left, focused detail panel on the right. Matches the third discovery PNG. Detail panel surfaces the diagnostic ("Hidden by preset: Hide hex color codes") plus per-tag actions: Always Show, Add to Hide List, Alias, Rename via Tag Wrangler. Cleaner for users who triage one tag at a time.

ProsExcellent per-tag affordances. Diagnostic is unmissable. Sets up well for v0.3 alias workflow.
ConsNeeds more horizontal room. Right panel awkward in narrow side leafs. More CSS to maintain.

Rule editor · Option 6

Three-pane builder

Rule list on the left, editor in the middle, live preview on the right. Matches the second discovery PNG. The live preview shows exactly which tags the rule currently hits, including a regex-safety hint inline. This is the most expressive surface; it replaces the modal in Task 13 if you want the polished path.

ProsLive preview prevents accidental hide-all rules. Fastest rule iteration. Looks premium.
ConsNeeds its own view, not a modal. More state to manage. Wider min-width.

Diagnostics · Option 7

Status bar variants

The status bar is the most-seen Tag Curator surface. Spec §8.3 calls it out specifically as the "where did my tag go" prevention. Here are four polished phrasings paired with different click-affordances. Pick one; do not stack them. The v0.1.0 plan ships variant B by default.

ProsAlways visible. One click opens the hidden-tag filter. Conveys plugin state without occupying space elsewhere.
ConsStatus bar text is constrained. Mobile lacks status bar - this surface is desktop-only.

Recommendation: Ship variant B as the default. It carries the count (the spec's required affordance) plus a click hint, uses the visible-dot pattern Obsidian users already recognize from indexing-status indicators, and degrades cleanly to variant C in dry-run and variant D when disabled. Variant A reads abruptly; variant E packs too much for an 11px status surface.
